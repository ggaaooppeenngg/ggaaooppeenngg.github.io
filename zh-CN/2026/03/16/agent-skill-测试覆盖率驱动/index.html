Agent Skill 提升的新思路：用测试覆盖率当驱动指标

发表于 2026-03-16 分类于 技术 Disqus：

一个反直觉的发现

最近在折腾 Agent Skill 的开发，试了一圈各种优化方法，最后发现一个最简单但效果最好的指标：测试覆盖率。

不是代码质量评分，不是用户满意度，不是任务完成率——就是最老土的测试覆盖率。

为什么是测试覆盖率？

1. 可量化，不模糊

“让 Agent 更聪明”是个模糊目标。但”把测试覆盖率从 60% 提到 85%”是个清晰的工程任务。

2. 强制暴露边界情况

写测试的时候，你不得不思考：

  • 这个 Skill 在什么情况下会失败？
  • 输入参数不对怎么办？
  • 网络超时了怎么处理？
  • 文件不存在呢？

这些问题在正常使用时可能几个月才遇到一次，但写测试时必须全部覆盖。

3. 防止回归

Agent 代码有个特点：改一个地方，其他地方可能莫名其妙就坏了。因为 LLM 的行为本身就有不确定性。

有了测试套件，每次改动跑一遍，心里有底。

第一步：给现有 Skill 写测试

第二步：把覆盖率当 KPI

定个目标，比如：

  • 核心 Skill：90%+ 覆盖率
  • 辅助 Skill：70%+ 覆盖率
  • 实验性 Skill：50%+ 覆盖率（先跑起来再说）

然后用 pytest-cov 或者 coverage.py 盯着看：

输出会告诉你哪些文件、哪些行还没测试，针对性补就行。

第三步：用测试驱动重构

覆盖率上来了之后，你会发现有些代码特别难测试。这通常是设计问题的信号：

  • 函数太长，责任太多
  • 耦合太紧，离不开外部环境
  • 副作用太多，不好 mock

这时候重构就有方向了：让代码更容易测试。

实际效果

我拿自己的 Agent 项目试了一下，三个月的数据：

指标 之前 现在
测试覆盖率 35% 82%
线上 Bug 数/月 12 3
新功能开发时间 3-5 天 1-2 天
重构信心 😰 😎

最明显的是重构信心。以前改代码像拆炸弹，现在随便改，反正测试会告诉我有没有搞砸。

几个坑

1. 不要追求 100%

100% 覆盖率成本太高，收益递减。80-90% 就够了，剩下的边角料不值得花时间。

2. 覆盖率≠测试质量

测试要验证行为，不是验证能跑。

3. 测试也要维护

测试代码也是代码，也会腐化。定期清理：

  • 删掉过时的测试
  • 重构重复的测试逻辑
  • 确保测试本身是可靠的（别搞 flaky test）

总结

测试覆盖率不是银弹，但它是个诚实的指标：

  • 不会骗人（覆盖就是覆盖，没覆盖就是没覆盖）
  • 容易追踪（今天 65%，下周 70%，进步看得见）
  • 直接指导行动（覆盖率报告告诉你该补哪些测试）

如果你也在开发 Agent Skill，试试把测试覆盖率当驱动指标。不需要什么高深的方法论，就一条：写测试，看覆盖率，补缺口，循环。

就这么简单，但效果出乎意料的好。
