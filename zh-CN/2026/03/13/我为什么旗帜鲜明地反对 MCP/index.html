我为什么旗帜鲜明地反对 MCP

发表于 2026-03-13 Disqus：

本文由 我的小龙虾 整理发布

开篇亮剑

先说结论：MCP（Model Context Protocol）是写给 LLM 的语言，不是写给机器的语言。

更准确地说：MCP 是一种僵化的工具协议——它既没有可被编程的属性，又缺少泛化能力。而脚本，恰恰相反。

今天这篇文章，我要旗帜鲜明地反对 MCP 的滥用。不是反对协议本身，而是反对那种”万物皆 MCP”的设计思路。

一、LLM 的核心能力是什么

1.1 代码能力是基模天赋

现代大语言模型（Claude、GPT-4、Qwen 等）在预训练阶段学习了海量代码：

  • GitHub 上的开源项目
  • Stack Overflow 的问答
  • 各种语言的文档和教程

结果是什么？ LLM 能：

  • 读懂 SDK 文档
  • 理解命令行 --help
  • 根据示例代码编写新脚本
  • 用管道组合多个工具

这是基模训练出来的天赋，不是 MCP 能赋予的。

MCP 工具是黑盒，脚本是白盒。

三、Anthropic Skills 把脚本推向了新高度

3.1 Skills 是什么

2026 年初，Anthropic 推出了 Skills 功能：

核心设计：

  • 用户编写脚本（Shell、Python 等）
  • 脚本放在指定目录（~/skills/）
  • Claude 自动发现并学习脚本的 --help
  • Claude 可以调用、组合、甚至修改脚本

官方描述：

“Skills are scripts that Claude can read, understand, and execute. They’re like tools, but programmable.”

3.2 Skills vs MCP

维度 MCP Anthropic Skills
定义方式 JSON schema 脚本文件
上下文加载 预加载全部 按需读取 help
可扩展性 需要注册新工具 直接写新脚本
可组合性 LLM 多轮调度 管道 `
可修改性 不能改实现 可以改脚本
泛化能力 低（预定义边界） 高（无限组合）

Skills 的本质： 把脚本的一等公民地位还给了 LLM。

6.3 原则三：输出不必是结构化，人类友好反而对 LLM 更友好

传统观点：

“LLM 需要结构化输出（JSON），否则解析困难”

这是错的。

脚本可以输出带颜色、带格式的人类友好内容：

为什么这对 LLM 更友好？

  1. 信息密度高 — 一眼看到关键信息（✓/✗/⚠）
  2. 无需解析 — LLM 直接读文本，不用 parse JSON
  3. 更精炼 — 不需要嵌套结构、字段名、引号

对比：

MCP 结构化输出 脚本人类友好输出
{"status":"success","failed":3,"warnings":2} ✓ OK ✗ 3 failed ⚠ 2 warnings
需要解析 JSON 直接读文本
字段名占 token 只有核心信息
嵌套结构复杂 一行搞定

LLM 理解哪种更容易？ 显然是后者。

实战建议：

  • 用颜色标记状态（绿✓ 红✗ 黄⚠）
  • 用表格展示数据
  • 用简短的文本摘要，不要嵌套 JSON
  • LLM 需要结构化时，可以自己 | jq 转换

6.4 原则四：脚本不要提供丰富参数

核心观点：

“脚本尽量不要提供丰富的参数，最好一个参数都不给，保证执行结果的确定性。”

为什么：

  • 参数越多，LLM 调用时越容易对参数值产生幻觉
  • 无参数脚本每次执行结果一致，便于调试和信任
  • AI 不需要理解参数含义，减少出错概率

七、总结

我反对的不是 MCP 协议本身，而是盲目崇拜 MCP、忽视脚本的设计倾向。

核心对比

维度 MCP 脚本
上下文 多（预加载 schema） 少（按需读取 help）
泛化性 低（预定义边界） 高（无限组合）
可被编程 否（黑盒） 是（白盒）
确定性 中（LLM 随机性） 高（代码固定）
LLM 参与度 被动使用 主动创造

Agent 工具设计四原则

  1. 上下文少优先 — 按需加载，不要预加载
  2. 泛化性优先 — 让 LLM 能自己组合、扩展
  3. 可被编程优先 — 白盒 > 黑盒
  4. 确定性优先 — 脚本执行结果可预测

最后

Anthropic Skills 的推出，把脚本模式推向了新高度。这不是巧合——脚本的可编程性和泛化性，恰恰是 LLM 最需要的。

LLM 有强大的代码能力，它能：

  • 读懂你的 SDK 和 help
  • 基于现有脚本编写新功能
  • 用管道组合多个脚本
  • 甚至修改和优化脚本

而 MCP，既没有可被编程的属性，又缺少泛化性。

脚本上下文少，泛化性高，何乐而不为？
