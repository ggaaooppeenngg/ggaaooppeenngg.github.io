DeepSeek-V4 Indexer 解析：FP4 量化的稀疏索引

发表于 2026-06-11 分类于 技术 Disqus：

前言

DeepSeek-V4 在 MoE 架构上引入了一个新组件：Indexer。它负责在 expert 路由之前，用 FP4 量化的轻量模型快速筛选候选 expert，减少后续 GEMM 的计算量。

SGLang PR #26209 进一步给 Indexer 的 Q/K 路径加了 NVFP4（E2M1 + UE8M0）量化，在精度损失极小（GSM8K 96.5% vs FP8 98.0%）的情况下，提速 1.08×–1.53×，KV-cache 压缩到 68 B/token（相比 FP8 的 132 B/token 省 ~48%）。

本文基于 PR #26209 的源码，从「数据格式 → Q 路径 → K 路径 → KV-cache 布局 → 上层调度」五层拆解。

一、Indexer 的定位

在 DeepSeek-V3 中，gating 网络直接输出 top-k expert 选择，计算量是 O(num_tokens × num_experts)。当 num_experts = 256 时，这个计算本身就很重。

Indexer 的思路：用 FP4 量化的小模型做「粗筛」，再用 BF16 做「精排」。

注意：只有 indexer 这条 sparse 路径（给 attention 选 top-k token 用的辅助打分）被换成了 FP4。主 attention 的 KV cache 完全没动，精度损失也只来自 indexer 的打分质量。

二、FP4 量化方案：NVFP4 (E2M1 + UE8M0)

PR 选用的不是普通 FP4，而是 Blackwell GenAI 推的 NVFP4 风格。

2.1 数据格式

  • 元素格式：E2M1（1 sign + 2 exp + 1 mantissa，共 4 bit）

    • 表示集合（绝对值）：{0, 0.5, 1, 1.5, 2, 3, 4, 6}
    • 最大正值：6.0，最小正规格：0.25

  • Micro-scale：每 32 个元素共享一个 UE8M0（8 bit 纯指数，无尾数）缩放因子

    • head_dim = 128 ⇒ 每 token 4 个 groups ⇒ 4 × 8bit scale = 32bit packed
    • UE8M0 表示 2^exp，无尾数，计算高效

2.3 反量化

三、Q 路径：fused_q_indexer_rope_hadamard_fp4_quant

文件：jit_kernel/csrc/deepseek_v4/main_norm_rope.cuh:658

一个 warp 处理一个 (token, head)，head_dim=128，vec=4（每 lane 4 个元素）。

3.1 流水线

stage 内容
① load 每 lane 一个 AlignedVector<DType,4>，bf16/fp16 输入
② RoPE 仅 is_rope_lane（末尾 16 lane，kRopeDim=64）做复数旋转
③ Hadamard-128 先 4×4 内寄存器蝶形（2 步），再 5 步 __shfl_xor_sync 跨 lane 蝶形，最后乘 1/√128
④ FP4 量化 warp::reduce_max<8> 求 group amax → scale = max/6 → cast_to_ue8m0 → quant_fp4_e2m1
⑤ store 4 个 fp4 → 16 bit 写 q_fp4；每 8 lane 一个 byte 写 q_sf（packed_sf 32bit）

3.2 关键技巧

  • reduce_max<8> 而不是全 warp：group size = 32 元素 = 8 lanes，正好和 NVFP4 的 micro-scale 粒度匹配
  • inv_scale_ue8m0 直接 bitcast 构造 fp32（scale_exp << 23），避免 1/scale
  • PDL（Programmatic Dependent Launch）：Hopper 特性，允许相邻 kernel 重叠
  • Hadamard 在量化之前：利用 Hadamard 的能量平均性，把 outlier 摊薄到 128 个元素，让 NVFP4 的窄动态范围（±6）够用

7.1 哪些张量是 FP4

张量 格式 由谁产生 由谁消费

logits 输出 bf16/fp32 (MQA scores) DeepGEMM SGLang sparse selector

7.2 主体在哪里？

SGLang（本 PR ~95% 代码量）：

  • 造数据：把 bf16 的 Q/K 量化成 (E2M1, UE8M0) 格式（2 个新 CUDA kernel + 1 套 Triton）
  • 存数据：KV-pool 按 68B/token、values/scales 分段排好
  • 接数据：在 forward_c4_indexer 里把 (q_fp4, q_sf) 和 68B page 喂给 DeepGEMM

DeepGEMM（零代码改动）：

  • 本 PR 没有改动 DeepGEMM 仓库，只是 from deep_gemm import fp8_fp4_paged_mqa_logits
  • DeepGEMM 早就支持读 (FP4 Q, FP4 K) 这条接口，SGLang 这次只是把数据按照它要的 layout 凑出来

一句话总结：FP4 的「格式定义、Q/K 量化 kernel、KV cache layout、数据流改造」主体都在 SGLang；DeepGEMM 只负责消费这一对 (FP4 Q, FP4 K) 计算 paged MQA logits。

八、设计亮点与潜在坑

亮点

  1. NVFP4 = E2M1 + UE8M0 (group=32) 是 Blackwell tensorcore 原生支持的格式，直接为后续 SM100 路径铺路
  2. Q 和 K 各自一个完全融合的 CUDA kernel：Norm/RoPE/Hadamard/量化/写 cache 全部在一个 warp 的寄存器里完成，输入读 1 次、输出写 1 次
  3. Hadamard 在量化之前：把 outlier 摊薄到 128 个元素，让 NVFP4 的窄动态范围（±6）够用（GSM8K 只掉 1.5pt 的核心原因）
  4. bitcast 构造 inv_scale（exp << 23）避免 fdiv；ceil_ue8m0 直接位运算避免 log/exp
  5. Page layout 把 values 和 scales 分两段连续摆放，匹配 DeepGEMM 的访存假设

潜在限制

  1. head_dim 强约束 128（assert x.shape[-1] == 128），group=32 → 4 个 UE8M0 → 正好 32bit
  2. page_size 必须是 2 的幂（K kernel 用 out_loc >> kPageBits）
  3. HIP 不支持 Q 融合 fp4 路径（AMD 端目前只能跑老 FP8）
  4. TileLang / AIter indexer 路径互斥，只能 DeepGEMM
  5. 4-bit 的低精度对长上下文/数值病态 prompt 仍可能发散，这就是它做成 opt-in flag 的原因

九、总结

Indexer 是 DeepSeek-V4 在 MoE 路由上的新尝试——用 FP4 量化的轻量模型做粗筛，减少后续 gating 和 expert GEMM 的计算量。

SGLang PR #26209 的完整实现是一个典型的「上游做格式+内存，下游做矩阵乘」的分工：

  • SGLang：格式定义、量化 kernel、KV-cache 布局、数据流改造（~95% 代码量）
  • DeepGEMM：消费 (FP4 Q, FP4 K) 计算 paged MQA logits（接口已存在，本 PR 没动）

核心创新不是 FP4 本身，而是 「Hadamard 摊薄 outlier + NVFP4 窄动态范围 + 全融合 kernel」 的组合，让 4-bit 量化在 indexer 这种「辅助打分」场景下的精度损失可控（1.5pt），换来了显存和速度的双重收益。
