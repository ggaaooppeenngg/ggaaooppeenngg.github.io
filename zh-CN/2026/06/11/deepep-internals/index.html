DeepEP 源码深度解析：MoE 专有的高效通信库

发表于 2026-06-11 分类于 技术 Disqus：

前言

DeepSeek-V3 之所以能在 2048 H800 上跑 671B MoE 模型，很大程度上归功于 DeepEP——一个专为 Mixture-of-Experts（MoE）设计的通信库。它不只是一个「更快的 NCCL」，而是把 MoE 的领域知识 hard-code 进通信原语，从根本上改变了「all-to-all 稀疏通信」的实现方式。

本文基于 DeepEP V1（legacy）源码，从 Python 接口到 CUDA kernel，逐层拆解 dispatch 和 combine 的实现原理。所有代码引用均对照 csrc/kernels/legacy/internode.cu 本地 clone。

一、背景：MoE EP 的通信痛点

MoE 模型中，每个 token 通过 gating 网络路由到不同的 expert，expert 分布在不同的 GPU 上。这个「动态、稀疏、不定长」的路由模式，暴露了传统 NCCL All-to-All 的三个硬伤：

  1. CPU 同步阻塞：NCCL 的 input_split_sizes 必须 .cpu()，导致 CUDA Graph 难以捕获
  2. 冗余流量：一个 token 选了同节点的 8 个 expert，NCCL 会发 8 份独立的 RDMA 流量
  3. 无法流式处理：NCCL kernel 是「批量语义」（等所有数据到齐才返回），没法在「RDMA 数据到达一半」时立即转发到 NVLink

DeepEP 的核心思路：把 MoE 的通信模式从「通用 collective」降级为「专用流水线」。

这样 RDMA QP 数量 = num_rdma_ranks（8 个节点 × 1 条/节点 = 8 条），不是 64 条。

4.2 元数据交换流程

  1. 本节点 NVL barrier：8 张卡通过 IPC 互写 per_rank_buffer，交换「每张卡分别要发给谁多少 token」
  2. 组装 RDMA metadata：汇总到 rdma_recv_num_tokens_mixed.send_buffer（41 ints/dst_rdma_rank）
  3. RDMA put + barrier：通过 nvshmem_int_put_nbi 一次性发到 8 个远端节点
  4. 散到本节点 8 卡：把远端元信息通过 IPC 写到本机各卡的 nvl_send_num_tokens_per_rank
  5. Channel 级前缀和：8 个 SM 各算一个 dst_rdma_rank，输出 rdma_channel_prefix_matrix 和 gbl_channel_prefix_matrix

4.3 为什么需要两次 barrier？

第一次确保所有卡写完 per_rank_buffer，第二次确保所有节点的 send_buffer 都 put 完了。

五、Step 3：dispatch kernel —— 20 SM 流水线

这是 DeepEP 最核心的部分。20 SM × 16 warp × 32 thread = 10240 个线程并发。

5.1 偶 SM / 奇 SM 对偶模型

关键洞察：偶 SM 和奇 SM 角色完全对称但方向相反——偶 SM 是「RDMA → NVL」的 forwarder，奇 SM 是「NVL → recv_x」的 receiver。

5.2 四类 warp 角色

SM 类型 warp 角色 数量 职责
偶 SM kRDMAAndNVLForwarder 8 从 RDMA recv buffer 读，forward 到 NVL recv buffer
偶 SM kForwarderCoordinator 1 反向流控：向远端发 RDMA put 更新 head
奇 SM kRDMASender 7 写 RDMA send buffer（有 sequential lock 保序）
奇 SM kRDMASenderCoordinator 1 凑齐 28 个 token 后发 RDMA put（三连击）
奇 SM kNVLReceivers 8 从 NVL recv buffer 读，写 recv_x（按 rank 重排）

5.3 四个环形队列

  • ① rdma_send_buf：capacity = 140，sender 侧用 sequential lock 保证 token 顺序
  • ② rdma_recv_buf：capacity = 140，forwarder 侧用 ld.acquire.sys.global 轮询 tail
  • ③ nvl_recv_buf：capacity = 288，receiver 侧用 gbl_channel_prefix_matrix 重排到 recv_x

5.4 三连击 RDMA 优化

为什么不直接用 nvshmem_int_put + barrier？因为 barrier 是同步阻塞调用。三连击让发送线程完全不阻塞——③ 是 fire-and-forget，远端通过 signal 自己触发消费。

5.5 常量来源

  • num_max_rdma_chunked_send_tokens = 28：28 × 7472 B ≈ 203 KB，是 InfiniBand HDR MTU 的整数倍，且 140 / 28 = 5 整除保证 wraparound 安全
  • num_max_rdma_chunked_recv_tokens = 140：必须是 28 的整数倍，让环形队列的模运算变成 & (capacity - 1) 的位运算
  • num_max_nvl_chunked_recv_tokens = 288：NVL 带宽更高，需要更深的 buffer 吸收 burst

handle 里的 send_nvl_head 和 send_rdma_head 是 反向地址簿——combine 时直接读这些数组，就知道「我应该写到哪个槽位」，绕开了所有协商开销。

七、Combine：dispatch 的逆运算

combine 是「多个 expert 输出 → 一个 token」的归约，有两个关键差异：

  1. 多了 reduce：用 atomicAdd 加权求和（权重来自 topk_weights）
  2. 没有 notify：所有 metadata 从 dispatch handle 复用

数据流（和 dispatch 反向）

SM 角色反转：dispatch 偶 SM 是 forwarder（往里收），combine 奇 SM 是 forwarder（往外发）。这个对称性让 buffer 布局可以复用。

八、如果没有 DeepEP，会怎样？

历史上 MoE 用 torch.distributed.all_to_all_single + NCCL。对比：

维度 NCCL all_to_all DeepEP normal DeepEP low-latency
单次 dispatch 时延 800-1200 µs 250-300 µs 80-120 µs
RDMA 带宽利用率 ~40% ~85% ~90%
NVL 带宽利用率 ~50% ~90% ~80%
SM 占用 32-128（波动） 20 固定 24 固定
CPU sync 次数 ≥2 1（cached=0） 0
Kernel 数 8+ 3 1

NCCL 的 6 个具体问题：

  1. CPU 同步阻塞 graph：至少 2 次 .cpu() 同步，CUDA Graph 难做
  2. 多次启动 kernel：layout + permute + 4× all_to_all + unpermute = 8+ kernel
  3. 多余拷贝：scatter_permute + NCCL 内部 2 次 + NCCL → recv_x = 4+ 次
  4. 没有节点内 forward 优化：一个 token 要去节点 j 的 8 张卡 = 8 份独立 RDMA 流量
  5. SM 占用不可控：NCCL kernel 是 collective，要和计算的 SM 抢
  6. Overlap 困难：NCCL 的 group_start/group_end 同步原语脆弱

九、DeepEP 的工程价值

不是「造个 NCCL 替代品」，而是把 MoE all-to-all 的领域知识 hard-code 进通信库：

  1. 去重投递：同节点多个 expert 只发一份 RDMA
  2. FP8 + scale 同步路由：layout 一次算，三个 buffer 同步搬
  3. handle 复用：decode 时 layout 不变就跳过 metadata 交换
  4. 精确 SM 预算：20 SM 是承诺值，可以放心规划其它 SM 给 GEMM
  5. CUDA Graph 原生兼容：low-latency 路径用 fixed layout + zero-init + 符号位 sentinel

DeepEP 的成功不在于「通信库本身有多快」，而在于它把 MoE 的通信模式从「通用 collective」降级为「专用流水线」，用领域知识换掉了通用抽象的开销。

十、总结

DeepEP V1 通过三阶段 pipeline（layout 计算 → metadata 交换 → 流水线搬数据），把 MoE 的动态稀疏通信从 NCCL 的「批量语义」改成「流式语义」。其核心创新不是某个单独的优化，而是整个通信栈的领域专用重构——从 RDMA 三连击到 NVL IPC 环形队列，从 SM 对偶模型到 handle 反向地址簿，每一层都在为 MoE 的通信模式量身定制。

V2 在此基础上进一步演进，用 JIT 编译 + ElasticBuffer 取代静态 channel 映射，让 SM 调度更灵活。但底层「RDMA-symmetric + NVL-IPC + head/tail 环形队列」的思想完全继承下来了。
