DeepSeek-V4 mHC Pre-Block 融合 Kernel 详解

发表于 2026-06-22 更新于 2026-06-24 分类于 技术 ， LLM Infra Disqus：

引言

DeepSeek-V4 引入了 mHC（multi-Hidden Carrier，多隐状态载体） 架构：每个 token 的隐藏状态扩成 hc_mult 份并行 carrier，一起进入 transformer 层。mHC pre-block 是这个架构的关键预处理阶段，负责：

  1. 把所有 carrier 的隐藏状态通过一次线性投影生成混合参数（$\tilde{A}_l, \tilde{B}_l, \tilde{C}_l$）
  2. 同时完成 RMSNorm 的平方和计算（$\sum x^2$）

SGLang 在 AMD MI355X 上用 aiter 库 实现了这个融合 kernel：mhc_pre_gemm_sqrsum_kernel。本文基于论文 §2.2、§3.2、§3.5.2 和源码，逐层拆解它的设计。

1. mHC Pre-Block 的数学目标（§2.2）

Carrier 间混合的数学定义

每个 token 携带 hc_mult 个并行 carrier，形状为：

每过一层 transformer，carrier 之间需要混合。论文 §2.2 定义混合由三组参数控制：

参数 形状 角色
$\tilde{A}_l$ $\mathbb{R}^{hc_mult \times hc_mult}$ carrier 间双随机混合矩阵（Sinkhorn 归一化后）
$\tilde{B}_l$ $\mathbb{R}^{hc_mult}$ scale 向量
$\tilde{C}_l$ $\mathbb{R}^{hc_mult}$ base/bias 向量

参数总数 = $hc_mult^2 + 2 \cdot hc_mult = (hc_mult + 2) \cdot hc_mult$

这就是代码中 mix_hc = (2 + hc_mult) * hc_mult 的来源。

mHC 的核心设计是：不分开学三个矩阵，而是合成一个大矩阵，一次投影出 mix_hc 维向量，再按位置切分：

2. Flatten：多路残差拼接（§2.2）

论文层面：每个 token 携带 hc_mult 个并行残差流，混合矩阵作用于所有 carrier 的拼接表示。

代码层面：

形状从 [4096, 4, 7168] 摊平到 [4096, 28672]。

在 TileLang/HIP kernel 看来，K 维就是长度 28672 的一维，这是后面 split-K 切分的基础。

物理存储：carrier 还是分开存的（hc_mult 份各自连续的 H 长度），但 K 维线性扫描时一并 reduce，等价于”所有 carrier 一起进 GEMM”。

3. 融合 Kernel：一次 HBM 大读同时完成 GEMM + 平方和

数学定义

输入：

  • x：[N, hc_mult × H] bf16，即 hidden_states.flatten(1)
  • fn：[mix_hc, hc_mult × H] fp32，可学习权重矩阵 hc_pre_fn

输出：

  • out：[N, mix_hc] fp32 — 小 GEMM 结果
  • sqrsum：[N] fp32 — $\sum x^2$ 结果（RMSNorm 平方和）

公式：

$$
\text{out}[i, j] = \sum_{k=0}^{hc_mult \cdot H - 1} x[i,k] \cdot fn[j,k]
$$

$$
\text{sqrsum}[i] = \sum_{k=0}^{hc_mult \cdot H - 1} x[i,k]^2
$$

一次 kernel 调用同时完成：

  1. 小 GEMM：out = x @ fn.T
  2. RMSNorm 平方和：sqrsum = sum(x², dim=-1)

为什么融合？

  • 输入 x 太大：4096 × (4 × 7168) × 2B = 224 MiB
  • 两个 reduce 都沿 K 维：sqrsum 和 GEMM 都对最后一维做累加
  • 分开两个 kernel：要从 HBM 读 x 两次 = 448 MiB
  • 融合后只读一次：224 MiB
  • MI355X HBM3e 5.3 TB/s：节省 ~42 µs

TileLang 参考实现

关键融合点：同一份 x_frag 从 shared memory 读出来后，既喂给 sqrsum 累加，又喂给 GEMM 累加。x 这块 224 MiB 的大输入只从 HBM 读一次。

4. Split-K 分块多阶段归约（§3.5.2）

为什么需要 Split-K？

K 维 28672 太长，一个 block 不可能在一遍 wave 内 reduce 完。Split-K 的解决方案：把 K 切成 split_k 份并行 reduce，最后 deterministic 合并。

为什么不用 atomic add？

atomic 在 reduction 时不能保证浮点确定性（顺序依赖）。论文强调 “deterministic reduction” —— 用独立的 stage 1 kernel 按固定顺序合并 partial sums，结果 bit-exact 可复现（对调试、训练对齐很关键）。

两阶段实现

Stage 0：每个 split 独立算 partial 结果

Stage 1：deterministic 归约

按固定顺序累加（无 atomic），可重现。

Split-K 取舍

split_k M 轴并行度 每 block 工作量 scratch 显存 atomic?
1（无 split） N/token_block K 太大，tile 间空闲 SM 0 否
4 4×N/token_block 均衡 4× partial buffers 否（stage 1 合并）
8/16 更高 reduce 阶段开销变大 8×/16× partial buffers 否
K_chunks 极致并行 几乎纯 reduce 大 否

对 MI355X（256 CU）：split_k=4~8 是常见取舍，刚好让 stage 0 覆盖所有 CU。

5. TileLang 实现细节（§3.2, §3.5.2）

(a) Tile 配置

每个 block 处理一个 32 token × 256 K 的切片；扫完整个 K 需要 28672/256 = 112 次迭代。

(b) Shared Memory 双缓冲 + 流水

num_stages=2 → double buffering：当前 iter 计算时，下一个 iter 的 x、fn tile 已经在 prefetch 到 shared memory，完全隐藏 HBM 延迟。

(c) Swizzled Layout 避免 Bank Conflict

把 shared memory 里的 x_smem 按 swizzled 模式排，确保 Matrix Core / wave 同时读时 32 个 bank 都被均匀访问，0 bank conflict。

(d) clear_accum=False 的 GEMM

每次迭代不清零累加器，沿 K 维持续累加 —— Matrix Core 友好。

(e) PDL（Programmatic Dependent Launch）

让 stage 0 → stage 1 之间的依赖在 GPU 上自己处理，不走 CPU。整个 mhc_pre 流水线可以 CUDA-graph capture 一气呵成。

(f) fp32 Accumulation, bf16 Storage

  • x 存 bf16（占带宽小）
  • x_frag 在 register 里转 fp32 算（精度高）
  • sqrsum_part, out_frag 都是 fp32

→ bandwidth-friendly + 数值稳定的标准配方。

5.5 Split-K GEMM 详解：瘦 GEMM 的 GPU 占用问题

普通 GEMM 分块回顾

GPU 标准做法：把输出 C 切成 tile，每个 thread block 负责一个 tile：

关键约束：每个 block 沿 K 维独立 reduce，不需要和别人通信。

问题：K 很大但 M×N 很小，会怎样？

考虑 mHC 那个 GEMM：

用普通分块，tile 大小取 Mt=128, Nt=24：

MI355X 有 256 个 CU → 32 个 block 只能填 12.5% 的卡！

SM/CU 大量闲置。同时每个 block 要扫完整个 K=28672，工作量极大、延迟极高。

这就是典型的 “K-heavy / 瘦 GEMM”：M×N 不足以填满 GPU，K 反而是大头。

Split-K 思路：把 K 维也切，多个 block 协作算同一个 C tile

把 K 切成 split_k 段，每段长度 K/split_k。每个 (输出 tile, K 段) 都由一个 block 负责，算出”局部累加结果”，最后再 reduce：

block 数从 32 暴涨到 32 × 16 = 512，刚好填满 MI355X 的 256 个 CU（还有冗余）→ GPU 占用率 100%，单 block 工作量减为 1/16，延迟 ≈ 1/16。

Reduce Partial 结果：两种做法

split_k 个 partial 必须合并成最终 C_tile：

方法 A：Atomic Add Reduction

  • ✅ 不需要中间 buffer，省显存
  • ❌ 浮点 atomic 不确定：不同 block 的累加顺序每次都不一样，浮点不满足结合律，结果每次略有差异
  • ❌ 多个 block 同时写同一个地址 → 序列化，写带宽下降

方法 B：Separate Reduction Kernel（Split-K with Epilogue）

  • ✅ 完全确定性：固定顺序累加，bit-exact 可复现
  • ✅ 各 block 写不同地址，无 atomic 竞争
  • ❌ 需要 split_k × M × N × sizeof(fp32) 的中间 buffer
  • ❌ 多一次 kernel launch（但 reduce kernel 很轻量）

这就是 mHC mhc_pre_gemm_sqrsum 用的方式。论文 §3.5.2 强调的 “deterministic reduction” 指的就是方法 B。

什么时候用 Split-K？

定性：输出 tile 数填不满 GPU，但 K 很大时用。

定量判断：

最佳 split_k 还要满足：

  • K / split_k ≥ Kt × num_pipeline_stages（保证每 block 仍有足够 K 工作量）
  • split_k 不能太大，否则 reduce 成本超过收益
split_k M 轴并行度 每 block 工作量 scratch 显存 适用场景
1（无 split） N/Mt K 太大，tile 间空闲 0 M×N 足够大时
4 4×N/Mt 均衡 4× partial buffers 常见取舍
8/16 更高 reduce 阶段开销变大 8×/16× partial buffers MI355X 推荐
K_chunks 极致并行 几乎纯 reduce 大 K 特别大时

一句话总结：Split-K 是瘦 GEMM（K 大、M×N 小）的标准优化 —— 把 K 维也切分，让更多 block 协作，填满 GPU，同时用两阶段 deterministic reduce 保证数值可复现。

5.6 什么时候用 Split-K：判断标准

GEMM 形状 例子 用 split-K？
方阵 / M、N 都大 标准 Linear、attention QK 投影 ❌ 输出 tile 已经够多
瘦 GEMM（M 大，N 小，K 大） mHC pre-projection (4096×28672×24) ✅
矮 GEMM（M 小，N 大，K 大） speculative decode 的 draft GEMM ✅
极瘦（M=1，K 大） LLM decode 阶段的 batch=1 GEMV ✅ split-K 必备
小 GEMM（K 很小） Q×K 在长 head_dim 下也可能 ⚠️ 不一定，看 K 是否成瓶颈

LLM 推理 decode 阶段的 token-level GEMV 是 split-K 最重要的应用场景：每个 token 一次 forward，所有 Linear 都是 [1, hidden] @ [hidden, out]，M=1 → 普通分块只能用 1 个 block，必须 split-K 才能用满 GPU。

工程细节

  1. 中间 partial 用 fp32：A: bf16, B: bf16 → partial: fp32 → 最终 C: bf16/fp32。每个 partial 都是 K/split_k 个数累加，必须用 fp32 防精度损失。
  2. partial buffer 的 layout：scratch[split_k, M, N] — split_k 在最外层。stage 0 写时每个 split 写自己的连续片段（避免 false sharing），stage 1 读时按 [m, n] 位置依次累加 split 维。
  3. K-split 不一定等大：如果 K % split_k != 0，最后一个 split 工作量略小。一般实现会让前面 split 取 ceil(K/split_k)，最后一个吃尾巴，或者 padding 输入到对齐。
  4. Stream-K（更先进的变体）：NVIDIA cutlass 后来引入 Stream-K：动态负载均衡，每个 block 自己抢一段 K 工作直到做完，彻底消除”最后几个 block 落单”的尾延迟问题。AMD 这边 aiter/Composable Kernel 也有类似实现。

mHC 的具体配置验证

输出 tile 数（无 split-K）= ceil(4096/32) × 1 = 128 个 block
→ 256 CU 占 50%，不够。

加上 split-K=16：
block 数 = 128 × 16 = 2048
每个 block：处理 [32 tokens × 1792 K] 的工作
scratch：[16, 4096, 24] × 4 byte = 6 MiB（合理大小）
每 block K 工作量 ≈ 2.75 Mflop，理论可达性：2048 个 block 分配到 256 CU → 每 CU 8 个 block，pipeline 充足。

完美匹配 GPU 的并行结构。

7. 性能账本（hc_mult=4, N=4096, H=7168）

阶段 HBM 读 HBM 写 主要工作
① 融合 GEMM+sqrsum split-K x: 224 MiB + fn: 2.6 MiB partial out: ~1.5 MiB, partial sqrsum: ~64 KiB bandwidth-bound
② deterministic reduce partials: ~1.5 MiB out + sqrsum: ~400 KiB 极轻量
③ rsqrt + scale sqrsum + out: ~400 KiB mixes: ~400 KiB 极轻量
④ Sinkhorn Ã: ~256 KiB comb: ~256 KiB iter 次数 × Sinkhorn 一次

第 ① 步是绝对热点，占整个 mHC pre 90%+ 时间。

不融合的话需要：

  • 单独 GEMM kernel：读 x 一次（224 MiB）
  • 单独 sqrsum kernel：再读 x 一次（224 MiB）

融合节省了 224 MiB 的重复读 → ~42 µs（按 5.3 TB/s 算）。在 61 层 × 每层都执行的累积下，效果显著。

9. mHC Pre-Block 第二阶段：mhc_pre_big_fuse_kernel

一句话说明

mhc_pre_gemm_sqrsum_kernel 算出的 partial 结果（每 token 的 sqrsum 和 mixes）作为输入，mhc_pre_big_fuse_kernel 在一个 kernel 里完成：split-K reduce → RMSNorm 归一化 → 三组参数激活 → Sinkhorn 迭代 → residual 加权合并，直接输出可以喂给 attention 层的 layer_input。

模板参数解码

Kernel 内部 6 步流水

每个 block 负责一个 token（grid = num_tokens），block 内不同 warp 干不同的事。

Step 1：Split-K 确定性归约

固定循环顺序，不用 atomic，bit-exact 可复现（论文 §3.5.2）。

Step 2：RMSNorm rsqrt + 应用归一化

数学等价于”先把 x 归一化再投影”（线性算子的可交换性），但工程上先 GEMM 后归一化更省 HBM 访问。

cm ∈ ℝ^{hc_mult × hc_mult} 是 $\tilde{A}_l$，需变成行和=1 且列和=1 的双随机矩阵：

这就是论文里的 $A_l$（carrier 间双随机混合矩阵）。

Step 5：用 pre_mix 把 carrier 加权合并成单流 layer_input

按 hidden_block=256 切片 pipeline，fp32 累加 → bf16 写出。

Step 6：Warp 级工作分配（Warp Specialization）

两个任务在 block 内同时执行，总时长 ≈ max(两者)，不是 sum。这是论文 §3.2 “fused kernel” 的精髓。

两阶段完整 Pipeline 串联

性能账本（hc_mult=4, N=4096, H=7168, n_splits=16）

阶段 HBM 读 HBM 写 瓶颈
① GEMM + sqrsum split-K x: 224 MiB + fn: 2.6 MiB partials: ~1.5 MiB bandwidth-bound
② big fuse partials: ~1.5 MiB + residual: 224 MiB layer_input: 56 MiB + comb: ~256 KiB mixed
合计 ~226 MiB ~57 MiB

不融合的话需要 6+ 个独立 kernel，每次都要重读 residual（224 MiB）。融合后省掉 5 个 kernel launch（~25 µs）+ 中间 buffer 反复写回 HBM。

一句话总结：mhc_pre_big_fuse_kernel = 把 split-K reduce + RMSNorm + 参数激活 + Sinkhorn + carrier 合并全部塞进 1 个 kernel，用 warp specialization 让计算密集（Sinkhorn）和 bandwidth-bound（carrier 合并）的任务并行跑。
