如何分析和阅读 DeepSeek-V4 的 Trace

发表于 2026-06-22 分类于 技术 ， LLM Infra Disqus：

引言

Perfetto 是 Android 平台主流的 trace 分析工具，近年来也被广泛用于 LLM 推理性能分析。它的优势在于：

  • 端到端视图：CPU 调度、GPU kernel、内存操作、跨进程通信全部在一个时间轴上
  • 交互式分析：框选、聚合、SQL 查询，适合 exploratory profiling
  • 离线可用：ui.perfetto.dev 是纯前端，trace 文件不离开本地

DeepSeek-V4 系列模型（MoE 架构、MLA attention、FP8 量化）的推理过程涉及大量 GPU kernel launch、集合通信（DeepEP）、KV cache 管理，trace 模式非常典型。本文以 DeepSeek-V4-Pro 推理为例，演示如何用 Perfetto 定位性能瓶颈。

1. 打开 Trace 与基础操作

加载方式

方式 1：直接用 Perfetto UI（推荐）

  1. 打开 https://ui.perfetto.dev
  2. 点击左侧 “Open trace file”（或直接把文件拖进浏览器窗口）
  3. 选择 .trace.json.gz 文件 —— Perfetto 原生支持 gzip 压缩，不需要先解压
  4. 等待加载（大文件可能要几十秒到几分钟）

方式 2：文件太大时启动本地 trace_processor

然后打开 https://ui.perfetto.dev，它会自动检测到本地的 localhost:9001 后端并使用它。

核心快捷键

快捷键 功能
W / S 水平缩放
A / D 左右平移
M 标记选中区间
F 放大到选中区间
Ctrl+F 搜索 slice 名称
框选 选中一段时间 → 底部显示聚合统计

2. 识别 DeepSeek-V4 推理的 Trace 特征

Prefill vs Decode：梳子图案

DeepSeek-V4 推理的 trace 在时间轴上呈现非常明显的模式：

  • Prefill 阶段：一段较长、kernel 密集且不规则的区域（处理多条请求的 input tokens）
  • Decode 阶段：严格周期性的”梳子”图案 —— 密集等宽的条带，每个齿 = 一个 decode step

识别技巧：先缩到最远（按 0）看整体节奏，先认出梳子，再放大数齿。第 N 齿到第 N+1 齿就是一个完整 decode step 的时长。

用 Kernel Launch 参数反推 Token 数

在 trace 里看到一个 vectorized_gather_kernel：

推算过程：

  • 总线程数 = 4096 × 4 × 256 = 4,194,304
  • 每个线程拷贝 16 字节 → 本次 kernel 共搬 64 MiB
  • grid.y = 4 对应 hidden_dim 切分：ceil(7168 × 2B / (256 × 16B)) = 4

结论：token 数 = 4096，这对应一次较大的 prefill batch（比如 1 条 4096 token 的 prompt，或多条总和 4096 tokens）。

DeepSeek-V4-Pro 配置：hidden_size = 7168，vocab_size = 129280，bf16 = 2 bytes

锚点 Slice 搜索

在搜索框（Ctrl+F）里搜以下关键词，快速定位 decode step 边界：

关键词 对应框架
forward_decode / ModelRunner.forward SGLang
executor_step / generation_step TRT-LLM
ncclAllReduce / ncclAllGather NCCL 通信（TP 场景的天然节拍器）

3. GPU vs CPU：事件归属判断

看 Track 名称（最直接）

Track 名包含 执行位置
GPU 0 / cuda / stream <N> / HIP stream GPU 上真实执行
python / MainThread / 进程名 / pid <N> CPU 线程
kernel（在 GPU track 下） GPU kernel

PyTorch profiler / SGLang 通常产出：

  • CPU tracks：python (pid xxx)、MainThread、各个 worker thread
  • GPU tracks：stream 7、stream 13 等（CUDA stream），或 GPU 0

Slice 名称特征

特征 说明
aten::xxx PyTorch op 调度（CPU 侧）
cudaLaunchKernel / hipLaunchKernel launch 动作本身在 CPU
ampere_sgemm_... / flash_fwd_kernel GPU kernel 真实名称
ncclKernel_AllReduce_... NCCL 集合通信 kernel

Flow Events：从 CPU Launch 到 GPU Kernel

Flow Events 是 Perfetto 里连接两个 slice 的”流向箭头”，在 GPU profiling 中非常重要：

这根线告诉你 CPU 在哪里发起了 launch，GPU 上对应的 kernel 是哪个。点 CPU 上的 cudaLaunchKernel slice，在右侧面板找到 flow，可以直接跳到 GPU 上的对应 kernel，量 launch latency。

4. 框选与聚合分析

Area Selection 组合拳

  1. 框选一段 decode step（鼠标左键拖动）
  2. 切到 Pivot Table，按 category 分组、按 SUM(dur) 排序 → 看 GPU kernel 总占比 vs CPU 占比
  3. 切到 Flamegraph → 找最宽的那根 → 确定瓶颈 op
  4. 切到 Slices，按 dur 降序 → 看具体哪几个 slice 最慢
  5. 点其中一个 CPU 上的 cudaLaunchKernel → 看 Flow Events → 跳到 GPU 上对应 kernel

SQL 查询速查

5. 实战：DSV4 两个关键函数的 Trace 表现

5.1 prepare_mlp_sync_batch

作用：在 DP/EP/CP 多 rank 协同推理时，把 batch 对齐成”所有 rank 都能一起做集合通信”的形状。

解决的问题：每个 attention DP rank 处理的 token 数可能不同，但 MLP/MoE 阶段的 all-gather / reduce-scatter / all-to-all 要求每个 rank 的 tensor 形状一致。

在 trace 中通常看到：

  • CPU 上的 Python overhead：ceil_align 循环、字典/列表操作
  • 小张量的 padding 操作 → 反复出现 CatArrayBatchedCopy kernel（对应 torch.cat 调用）
  • 如果耗时明显（几百 µs 起步），通常是 CPU-bound，优化方向是把 padding 融合或缓存

5.3 从 Trace 反推线程利用率（以 Embedding 为例）

在 trace 里看到一个 vectorized_gather_kernel<16, long> 的 launch 参数：

可以反推出两层信息：GPU 是否被填满，以及启动的线程里有多少真在干活。

线程账本

12.5% 的浪费来自 hidden=7168 不能被 block.x × 元素/线程 = 256 × 8 = 2048 整除。每行需要 ceil(7168/2048) = 4 个 block，第 4 个 block 只填了一半，余下线程 early-exit。

这个浪费比例与 N_tokens 无关，是 kernel launch 配置 × hidden_size 的固有特性。

GPU 是否被填满？

以 MI355X（256 个 CU）为例：

严重 oversubscribed，远超饱和点。4096 token 对 SM/CU 占用已经”管够”，再增大 token 数没有收益。

GPU 的真正瓶颈是 HBM 带宽：embedding lookup 是 random gather，从 1.85 GB 的权重表里抓 4096 行，cache 命中率极低。

如何让”启动的线程都干活”？

方案 每行 block 数 每行总线程 写入元素 浪费
现状 block=256, vec16 4 1024 7168/8192 12.5%
block=128, vec16 7 896 7168/7168 0% ✓
block=64, vec16 14 896 7168/7168 0% ✓

最优是 block.x=128（或 64），vec=16B：每行正好 7168 个 bf16，0 浪费。但 PyTorch 原生 vectorized_gather_kernel 是模板写死的，用户不能调。要消除这 12.5%，需要自己写 Triton/HIP kernel。

多少 Token 让 Launch Overhead 摊销最划算？

N_tokens 总 block 数 每 CU block 数 备注
1 4 0.016 launch overhead 占绝对大头
64 256 1 刚够每 CU 一个 block
256 1024 4 推荐下限
1024 4096 16 占用充足
4096 16384 64 占用饱和
16384 65536 256 收益边际递减

结论：对 MI355X + DSV4-Pro，N_tokens ≥ ~512 就足以把 embedding kernel 的 GPU 占用打满。

诊断建议：如果 trace 里 embedding kernel 占整个 forward step 时间 < 1%，不用管它；如果占比 > 5%，说明 step 整体太快，可以考虑把 embedding 和后续 RMSNorm 融合，避免 56 MiB 中间结果写回 HBM。

5.4 mHC 扩展：Embedding 之后的第一次 Copy

DeepSeek-V4 引入了 mHC（multi-Hidden Carrier，多隐状态载体） 结构：每个 token 不再只用一个 hidden vector 走完模型，而是复制成 hc_mult 份并行 hidden vector，一起进入后续 transformer 层。

代码位置

hc_mult 从模型 config 读取（典型值 2 或 4）。

在 Trace 里的表现

direct_copy_kernel 的来源：.repeat() 需要 .contiguous() 实物化数据 → 触发 aten::copy_。

数据量

launch config 通常是 block=128 × unroll 8，每 block 处理 1024 个元素，总 block 数 = N × hc_mult × hidden_size / 1024，连续访存，HBM 顺序读 + 顺序写，效率接近峰值带宽。

优化空间

方案 可行性
修改 VocabParallelEmbedding 直接输出 [N, hc_mult, H]，省掉 repeat 需要改 SGLang 源码
保留 expand 视图，下游层用 stride=0 张量计算 下游 GEMM 需要连续内存，不可行
延迟物化：让下游第一层直接 broadcast hc_mult 维度 mHC 设计要求每个 carrier 走独立子路径，无法 broadcast 共享

如果 trace 上看到这个 copy 在 30–50 µs 区间，已经是接近 HBM 峰值的最优形态，占整个 step 比例极小，通常不值得专门优化。

5.5 DP Gather 链：mHC 扩展之后的元数据准备

mHC 扩展 → prefix scan → DP gather 是 DSV4 forward 开头的第二条三连曲。

连续出现的 3 个 Kernel

在 trace 里，direct_copy_kernel<BFloat16>（mHC 复制）之后通常会连续出现：

① rocprim::scan<long> — int64 前缀和

对应代码（dp_attention.py:392）：

  • 输入：global_num_tokens_gpu，shape [dp_size]，int64，记录每个 DP rank 的 token 数
  • 输出：cumtokens[i] = sum(global_num_tokens[0..i])
  • 用途：本 rank 的全局起始偏移 = cumtokens[dp_rank - 1]
  • 数据量极小（仅 dp_size 个元素），launch overhead 比实际计算大得多

② FillFunctor<long> — int64 buffer 填零

用 “scatter-then-allreduce” 模式做 DP gather：每个 rank 只往”自己那段”写真实数据，其他段必须是 0，all-reduce 求和后才得到正确的全局拼接结果。

③ memcpy_triton_kernel — 带 GPU 端 offset 的拷贝

这是 SGLang 自己写的 Triton kernel，关键设计是 offset 和 size 都留在 GPU 上读：

普通 tensor[a:b] = src 需要 a, b 是 Python int → 必须 .item() → 强制 D2H 同步。Triton kernel 避免了这个同步，整条 DP gather 路径都能被 CUDA graph capture。

整体在干什么（一句话）

每个 DP rank 把自己的 input_ids 通过”zero-fill 全局 buffer → 写到自己的偏移位置 → all-reduce 求和”的方式，拼成所有 DP rank 的全局 input_ids 张量。这个全局张量后面 MoE / EP all-to-all 需要用到（每个 expert 要知道全局所有 token，才能决定路由）。

性能特征

kernel 耗时估算（MI355X, 4K tokens） 说明
rocprim::scan 3–5 µs launch overhead 占大头
FillFunctor<long> 5–15 µs 取决于 buffer 大小
memcpy_triton_kernel 5–10 µs 4096 × 8B
NCCL all-reduce 几十 µs ~ 几 ms 取决于 NIC

如果 trace 里这段总计 ~100 µs，基本是最优形态；如果远超，检查是否 DP size 过大或 NIC 有问题。

识别技巧：在 trace 里看到 rocprim::scan<long> + FillFunctor<long> + memcpy_triton_kernel 连续出现，可以直接判定进入了 DP gather 路径，这是 forward 开头的强路标。

SUM 模式 vs MAX_LEN 模式

模式 buffer 长度 通信原语 适用场景
SUM sum(global_num_tokens) All-Reduce rank 间 token 数差异不大，总 padding 少
MAX_LEN max(global_num_tokens) × dp_size All-Gather 部分 rank 远大于其他，避免重复传 0

DpPaddingMode.get_dp_padding_mode() 根据 is_extend_in_batch 和 global_num_tokens 的分布自动选择。

为什么在 GPU 上算 Offset？（CUDA Graph 兼容性）

一个关键设计决策：为什么不在 CPU 上算好 local_start_pos 再传给 GPU？

  1. CPU 算需要 .item() → D2H 同步 → 阻塞 GPU
  2. CUDA graph 不能 capture .item() → DP gather 路径无法 graph 化
  3. GPU 上 cumsum + Triton memcpy → 全异步 + 可 capture

代价是多一次 launch overhead（~5 µs cumsum + ~5 µs memcpy_triton），但让整个 DP gather 进 CUDA graph 的收益远大于这点 overhead（CUDA graph 节省每 step 几十到上百 µs 的 launch 时间）。

5.8 Attention 阶段：CK FP8 GEMM（kernel_gemm_xdl_cshuffle_v3）

紧接 mHC pre-block 完成之后，进入 attention 的 Q/K/V 投影。trace 里会出现一个很长的 mangled 名：

这是 Composable Kernel（CK） 的 FP8 GEMM —— AMD 版 CUTLASS，gfx950（MI355X）上 GEMM 的事实标准。

名字解码

片段 含义
ck:: Composable Kernel 库
kernel_gemm GEMM kernel 主体
xdl 使用 XDLOPS（gfx940/950 的 Matrix Core 指令）
cshuffle C-shuffle 输出 epilogue：在 shared memory 里重排输出再写回 HBM
blockscale 逐 block scale 量化支持（group_size=128 的 FP8 量化）
b_preshuffle B 矩阵（权重）预先重排成 kernel 最优访问模式

一句话：“在 gfx950 Matrix Core 上跑的 FP8 GEMM，支持 block-scale 量化，B 矩阵预排，输出经 C-shuffle 重排写回”。

模板参数含义

从 mangled 名提取的关键参数：

参数 值 含义
A layout RowMajor activation [M, K]，FP8
B layout ColumnMajor 权重 [K, N]，FP8（做了转置）
C layout RowMajor 输出 [M, N]，fp32/bf16
a_type f8_fnuz_t FP8 e4m3 FNUZ（AMD 专用 FP8 格式）
b_type f8_fnuz_t 同上
M_tile 128/256 每 block 输出 tile 的 M 维
N_tile 128/256 每 block 输出 tile 的 N 维
K_tile 128 每次 K 迭代步长
ScaleBlockM 128 A 的 scale 粒度（对应 group_size=128）

FNUZ = “Finite, No Unsigned Zero” —— AMD MI300 系列的特殊 FP8 编码，与 NVIDIA OCP FP8 略有不同。

B Preshuffle 是什么？

权重矩阵 B 在模型加载时就 offline 重排过一次，把元素按 kernel 最优访问模式重新排列：

代价是：B 的预排数据需要在加载权重时一次性计算，存到 HBM 里（布局已变）。SGLang 在模型 init 时通过 aiter.ops.shuffle.shuffle_weight 一类工具做这件事。

在 Trace 里的位置

为什么用 CK 而不是 aiter/Triton？

库 适用场景
aiter ROCm 上 LLM 推理的 fused kernel（RMSNorm、attention、MoE）
Composable Kernel AMD 官方 GEMM 库，支持 FP8 block-scale、灵活 epilogue
Triton 快速原型，性能通常不如前两者

DSV4 的 attention QKV 投影用 CK 是因为：FP8 block-scale GEMM + 自定义 epilogue（C-shuffle） 是 CK 的强项。

一句话：“DSV4 mHC 张量在进入 GEMM 之前，先做 RMSNorm 的平方和累加（融合版）”。

为什么拆成独立 Kernel？

  1. mHC 维度特殊处理：输入是 [N, hc_mult, hidden]，每个 (token, carrier) 对独立归一化，标准 RMSNorm kernel 不支持这种 3D layout
  2. HBM 带宽复用：input 从 HBM 读出来，顺便把 ∑x² 算了，下一个 kernel 不用再读一次（fused-prologue 模式）

模板参数含义

参数 值 含义
BLOCK 256 线程块大小
WAVE 64 AMD wavefront 大小（gfx950 wave64）
BYTES 32 每线程一次读多少字节（vectorized load）
TILE 128 hidden 维度切片大小

7168 / 128 = 56 恰好整除，这就是选 128 的原因。

aiter 是什么？

aiter（AI Tensor Engine for ROCm）是 AMD 开源的 LLM 推理 kernel 库，专为 MI300X / MI355X 手写优化，相当于 NVIDIA 阵营的 FlashAttention / FlashInfer。

SGLang 在 ROCm 上跑 DSV4 时，attention / MoE / norm 等性能关键路径会优先调用 aiter 实现，而不是 PyTorch 原生 op 或 Triton。

在 Trace 中的位置

识别技巧：mhc_pre_gemm_sqrsum_kernel 是进入 transformer layer 0 的强路标，看到它就说明 DP gather 已完成，正式进入计算阶段。

性能特征

MI355X HBM3e ~5.3 TB/s 理论：

  • hc_mult=2：~22 µs 理论
  • hc_mult=4：~44 µs 理论

实测如果远超，可能是模板参数不是最优实例化，或 HBM 被其他流抢带宽。

mHC 是 DSV4 专用特性

这个 kernel 只在 DSV4 上触发，LLaMA / Qwen / DSV3 不会走到这里 —— 因为它们的 hidden_states 是 2D [N, H]，不需要这种 3D layout 的 RMSNorm 预处理。

在 trace 里看到 aiter::mhc_* 系列 kernel，可以直接判定模型是 DSV4（或使用了 mHC 架构变体）。

下次在 trace 里看到 aiter::xxx，按下划线切开，基本能猜出它在干什么。

数学定义：一次 Kernel 同时完成 GEMM + 平方和

输入：

  • x：[N, hc_mult × H] bf16，即 hidden_states.flatten(1)，从 mHC 的 [N, hc_mult, H] 摊平
  • fn：[mix_hc, hc_mult × H] fp32，可学习权重矩阵（hc_pre_fn）

输出：

  • out：[N, mix_hc] fp32 — 小 GEMM 结果
  • sqrsum：[N] fp32 — ∑x² 结果（RMSNorm 平方和）

公式：

[
\text{out}[i, j] = \sum_{k=0}^{hc_mult \cdot H - 1} x[i,k] \cdot fn[j,k]
\qquad
\text{sqrsum}[i] = \sum_{k=0}^{hc_mult \cdot H - 1} x[i,k]^2
]

一次 kernel 调用同时完成：

  1. 小 GEMM：out = x @ fn.T，输出 shape [N, mix_hc]
  2. RMSNorm 平方和：sqrsum = sum(x², dim=-1)

后续如何使用这两个量（deepseek_v4.py:1133 参考实现）：

为什么融合：

  • 输入 x 太大：4096 × (4 × 7168) × 2B = 224 MiB
  • 两个 reduce 都沿 K 维：sqrsum 和 GEMM 都对最后一维做累加
  • 分开两个 kernel：要从 HBM 读 x 两次 = 448 MiB
  • 融合后只读一次：224 MiB
  • MI355X HBM3e 5.3 TB/s：节省 ~42 µs

模板参数对应 TileLang 实现的 tile 参数：

模板值 含义
256 hidden_block：K 维 tile 大小（hc_mult*H 维上切片）
64 AMD wave_size（wave64）
32 token_block：M 维 tile 大小（N 维上切片）
128 每 block 线程数

每个 block 处理 32 token × 256 hidden 的 tile，128 个线程协作，迭代 hc_mult*H / 256 次扫完一个 token block 的 K 维。

一句话总结：mhc_pre_gemm_sqrsum_kernel = 在 DSV4 mHC pre-block 里，把 mixes = x @ fnᵀ 这个小 GEMM 和 sqrsum = Σx² 这个 RMSNorm 平方和，融合在一次 224 MiB 的 HBM 大读里完成。

常见瓶颈模式

现象 可能原因
GPU stream 出现 gap，CPU track 密密麻麻 CPU bottleneck（Python overhead、kernel launch 慢）
