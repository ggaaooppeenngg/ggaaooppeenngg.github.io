DeepSeek-V4 Pro MoE 路由机制分析

DeepSeek-V4 Pro 使用了 Mixture of Experts（MoE） 架构，总计 384 个专家，每个 token 激活其中 6 个。其路由机制分为两种：Hash Routing（前 n 层） 和 Score Routing（后续层），这是理解其推理性能的关键。

两种路由模式对比

Hash Routing（前3层） Score Routing（后58层）
选择方式 tid2eid[token_id] 查表 scores.topk(6)
计算开销 O(1) 查表 [bs×seq, 7168] × [384, 7168]^T GEMM + topk 排序
是否可训练 否（requires_grad=False） 是
负载均衡 确定性均衡（预分配） 依赖 bias + auxiliary loss
语义依赖 不依赖（embedding 刚出来语义弱） 依赖（深层语义丰富）

为什么前几层用 Hash Routing？

核心原因：第 0 层输入刚从 embedding 出来，语义信息极弱，学习路由和随机/确定性分配效果差不多。

具体优势：

  1. 保证完美负载均衡 — 每个专家被分配到近似等量的 token，不会出现热门 expert 过载
  2. 消除路由抖动 — MoE 训练早期路由波动大，hash 直接消除这个问题
  3. 推理确定性 — 相同 input 永远走相同专家，便于缓存和优化
  4. 省去 topk 排序开销 — 前几层 router_gemm 的 GEMM + topk 直接省掉

tid2eid 是怎么生成的？

生成方式大概率是基于负载均衡算法（而非简单 hash）来确保所有专家被均匀分配。结果写入初始 checkpoint，requires_grad=False 保证全程不变。

总结

要点 说明
Hash Routing 前 n 层用查表代替学习路由，确定性、O(1)、负载均衡
Score Routing 深层用 top-k 学习路由，依赖语义信息
tid2eid 预训练前确定，全程固定（requires_grad=False）
可训练部分 只有 score 权重，决定各专家的贡献比例
设计动机 embedding 层语义弱，学习路由意义不大；固定映射更稳定、更均衡

这种设计体现了 DeepSeek 工程上的务实：不是所有层都需要学习路由，能用确定性解决的就不要浪费算力。
