DeepEP：为 MoE 而生的极致通信库

发表于 2026-05-15 分类于 技术 Disqus：

背景：MoE 的通信之痛

DeepSeek-V3/V4 这类大模型用了 **Expert Parallelism (EP)**：384 个专家分布在 8 张 GPU 上，每个 token 被路由到 6 个专家。

看起来很美好，但实际跑起来你会发现：

用标准的 NCCL all_to_all 算一笔账（H200，NVLink 900 GB/s）：

实际更糟：

  • 8 卡并发通信，带宽打 7 折
  • 突发小消息（每个专家只收到几十个 token）
  • 全局同步屏障，最慢的卡拖慢所有人

结果：通信占了 MOE 总时间的 77%。

DeepEP 是什么

DeepEP (DeepEveryParallel) 是 DeepSeek 开源的 MoE 专用通信库，专门解决上面的问题。

GitHub: https://github.com/deepseek-ai/DeepEP

最新 V2 版本做了完全重构，核心特点：

  • JIT 编译：运行时生成最优 CUDA kernel，无需安装时编译
  • NCCL Gin 后端：轻量级的通信后端，替代厚重的 NVSHMEM
  • Hybrid 模式：同时支持 NVLink（节点内）+ RDMA（跨节点）
  • 0-SM 通信：Engram/PP/CP 用 Copy Engine，不占计算资源
  • 分析性 SM 调度：自动算出最优 SM 数，不再需要 autotune

深入：V2 做了什么优化

1. JIT 编译（Just-In-Time）

传统做法：安装时编译所有 kernel，要支持 sm_90、sm_100 得编译多遍。

DeepEP V2：运行时根据参数生成最优 kernel：

好处：

  • 安装时无需知道目标 GPU 架构
  • 每次调用根据实际参数（num_ranks, num_topk, hidden）生成最优版本
  • 无需手动 autotune，自动算出最优 SM 数

2. NCCL Gin 后端（替代 NVSHMEM）

V1 用 NVSHMEM（NVIDIA 的 PGAS 库），依赖重、安装麻烦。

V2 换成 NCCL Gin（Header-only，轻量）：

优势：

  • Header-only，无额外依赖
  • 复用已有的 NCCL communicator
  • 性能持平或优于 NVSHMEM

4. 0-SM 通信（Engram, PP, CP）

传统 GPU 通信 kernel 需要占用 SM（Streaming Multiprocessor）来执行。

DeepEP V2 的 Engram/PP/CP 用 Copy Engine + RDMA：

源码中 engram.hpp 实现了这个机制，特别适合：

  • Engram：远程显存访问（类似 RDMA 的 get/put）
  • **PP (Pipeline Parallelism)**：流水线并行中的激活传递
  • **CP (Context Parallelism)**：长序列的上下文并行

总结

DeepEP 不是又一个 NCCL wrapper，它针对 MoE 的 不规则通信模式 做了深度优化：

  1. JIT 编译 — 运行时生成最优 kernel，适配任意 GPU 架构
  2. NCCL Gin — 轻量后端，替代厚重的 NVSHMEM
  3. Hybrid Dispatch — NVLink + RDMA 混合，跨节点无压力
  4. 0-SM 通信 — Copy Engine 干活，SM 全力算 GEMM
  5. 分析性调度 — 自动算 SM 数，告别 autotune
  6. FP8 支持 — dispatch 时量化，再砍一半通信量

如果你在跑 MoE 模型（DeepSeek-V3/V4、Mixtral、Qwen3-MoE），DeepEP 值得一试。
