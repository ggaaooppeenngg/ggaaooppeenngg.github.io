DeepSeek-V4 KV Cache 计算详解

DeepSeek-V4-Pro KV Cache Size Calculation

1. Overview

DeepSeek-V4 replaces V3’s MLA (Multi-head Latent Attention) with a hybrid attention
architecture combining CSA (Compressed Sparse Attention) and HCA (Heavily Compressed
Attention). Both compress KV cache along the sequence dimension, which is fundamentally
different from V3’s latent-dimension-only compression.

This document derives the KV cache size step by step from the model’s config.json and
reference implementation inference/model.py, cross-validated against the official technical
report and vLLM’s analysis.

2. Model Parameters

Source: config.json

Parameter Symbol Value
num_hidden_layers $L_{layers}$ 61
head_dim $c$ 512
qk_rope_head_dim $r$ 64 (included within $c$)
sliding_window $w$ 128
index_head_dim $c_I$ 128
index_topk $k$ 1024
compress_ratios 62-element array (see below)

2.1 Layer Classification from compress_ratios

The compress_ratios array has 62 elements: 61 for transformer layers + 1 for the MTP layer.

Type Count Layer indices
HCA (ratio=128) 31 0, 1, 3, 5, 7, …, 59
CSA (ratio=4) 30 2, 4, 6, 8, …, 60
Total 61

3. Key Architectural Difference from V3

V3 MLA: RoPE key is a separate 64-dim vector concatenated with the 512-dim latent.

V4 CSA/HCA: RoPE is applied to the last 64 dims of the 512-dim vector, in-place.

Evidence from model.py:

Conclusion: Each KV entry is 512 elements, not 576.

4. Precision Modes

The paper (Section 2.3.4) states:

“We adopt a mixed storage format for KV entries: BF16 precision is used for the rotary
positional embedding (RoPE) dimensions, while FP8 precision is applied to the
remaining dimensions.”

The paper (Section 5.2.1) further states:

“The Query-Key (QK) path in the indexer of CSA, where QK activations are
cached, loaded, and multiplied entirely in FP4.”

This gives us two precision configurations:

Symbol Meaning BF16 Mixed Precision Deployment
$S_{kv}$ Shared-KV per element bytes $c \times 2 = 1024$ $448 \times 1 + 64 \times 2 = 576$
$S_{idx}$ Indexer per element bytes $c_I \times 2 = 256$ $c_I \times 0.5 = 64$ (FP4)

5. KV Entry Structure (512 dims)

Comparison with pure BF16:

  • Pure BF16: 512 × 2 = 1024 bytes/entry
  • Mixed precision: 448 + 128 = 576 bytes/entry
  • Ratio: 576 / 1024 = 56.25% ≈ paper’s “roughly half”

6. HCA Layer (31 layers, ratio=128)

Only Shared-KV, no Indexer. After compression, entry count is small, so direct dense attention.

Per-layer cache = Compressed + Sliding Window

$$\text{PerHCA} = \underbrace{(\frac{N}{128} + w) \times c \times S_{kv}}_{\text{Shared-KV}}$$

Component Entries bytes/entry Subtotal
Compressed KV N/128 512 × 2 = 1024 (N/128) × 1024
Sliding Window 128 512 × 2 = 1024 128 × 1024
Total/layer $(\frac{N}{128} + 128) \times 1024$ bytes

HCA Flow

7. CSA Layer (30 layers, ratio=4)

Shared-KV + independent Indexer cache. After compression, entry count is large (N/4), so sparse selection is needed.

7a. Shared-KV Cache

Same structure as HCA, just different compression ratio:

Component Entries bytes/entry Subtotal
Compressed KV N/4 512 × 2 = 1024 (N/4) × 1024
Sliding Window 128 512 × 2 = 1024 128 × 1024

7b. Indexer Cache (CSA only)

Indexer has its own independent Compressor, independent wkv weights, independent cache buffer:

Component Entries bytes/entry Subtotal
Indexer Compressed KV N/4 128 × 2 = 256 (N/4) × 256

Note: Indexer has no sliding window. It only stores compressed entries for scoring.

7c. CSA Complete Flow

Key: Indexer’s 128-dim cache is only used for scoring (finding which 1024 entries are
most important). Actual attention’s key/value uses Shared-KV’s 512-dim entry.
Two caches serve different roles.

8. Total Formula

Let sequence length = $N$:

$$\text{Total KV} = \underbrace{N_{CSA} \times \text{PerCSA}}{\text{30 CSA layers}} + \underbrace{N{HCA} \times \text{PerHCA}}_{\text{31 HCA layers}}$$

Where:

$$\text{PerCSA} = \underbrace{(\frac{N}{4} + w) \times c \times S_{kv}}{\text{Shared-KV}} + \underbrace{\frac{N}{4} \times c_I \times S{idx}}_{\text{Indexer}}$$

$$\text{PerHCA} = \underbrace{(\frac{N}{128} + w) \times c \times S_{kv}}_{\text{Shared-KV only}}$$

11. Why So Small? Key Insights

  1. Sequence dimension compression (V3 only compresses latent dimension, V4 also compresses sequence length)

    • CSA: L → L/4
    • HCA: L → L/128

  2. Sparse attention (CSA only accesses top-1024 compressed blocks, not all L/4)

  3. Layered design: CSA handles precise mid-range dependencies, HCA provides global overview

  4. Mixed precision:

    • RoPE dims: BF16 (2 bytes, ensures positional precision)
    • Content dims: FP8 (1 byte, saves space)
    • Indexer: FP4 (0.5 byte, most aggressive)

  5. Disk KV Cache: Compressed KV blocks can be stored on disk, shared prefix reuse
